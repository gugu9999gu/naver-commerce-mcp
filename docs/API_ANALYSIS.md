# 네이버 커머스 API 상세 분석

기준 문서는 **커머스 API 2.80.0(2026-06-10)** 이다. 네이버 문서의 `current` 경로는 이후 업데이트될 수 있으므로, 운영 반영 전에 변경 이력을 다시 확인해야 한다.

## 1. 시스템 구조

커머스 API는 스마트스토어 기능을 서버 간 HTTP API로 공개한다. 기본 호스트는 다음과 같다.

```text
https://api.commerce.naver.com/external
```

일반 요청·응답은 JSON/UTF-8이다. 날짜와 시간은 ISO 8601을 따르며 KST 오프셋 `+09:00`을 사용한다. 장애 분석에는 응답 헤더 `GNCP-GW-Trace-ID`와 `GNCP-GW-HttpClient-ResponseTime`을 보존해야 한다.

## 2. 인증

OAuth 2.0 Client Credentials를 사용한다. 토큰 요청에서 `client_secret` 자체를 보내지 않고 전자서명을 보낸다.

```text
password = client_id + "_" + timestamp_ms
bcrypt_hash = bcrypt(password, salt=client_secret)
client_secret_sign = Base64(bcrypt_hash)
```

토큰 엔드포인트는 `POST /v1/oauth2/token`이다. `type=SELF`는 애플리케이션 자체 리소스, `type=SELLER`는 연결 판매자 리소스이며 SELLER는 `account_id`가 필요하다. 공식 문서상 토큰 유효 시간은 10,800초다. 동일 리소스의 기존 토큰이 30분 이상 남으면 기존 토큰이 반환되고, 30분 미만이면 새 토큰이 발급된다.

API 응답이 `401`이고 오류 코드가 `GW.AUTHN`이면 토큰 만료 가능성이 있으므로 토큰을 한 번 재발급한 뒤 원 요청을 한 번만 재시도한다.

## 3. 핵심 도메인

### 상품

원상품은 판매가·배송·상세 정보 등 공통 속성의 중심 객체다. 채널 상품은 스마트스토어 또는 쇼핑윈도 같은 채널별 속성을 표현한다. 상품 생성/수정 요청은 필드 수가 많고 카테고리, 상품정보제공고시, 원산지, 배송, 옵션 코드의 선행 조회가 필요하다.

그룹상품 등록·수정 같은 일부 API는 비동기다. 최초 HTTP 성공은 최종 반영 완료를 의미하지 않으므로 요청 결과 조회 API를 폴링해야 한다.

### 주문과 클레임

`orderId`와 `productOrderId`를 구분해야 한다. 실제 발주·발송·취소·반품·교환 작업은 주로 상품주문 단위다. 변경 상품 주문 조회는 증분 수집의 기준 API이며, 최대 300건 이후 `more` 커서를 이용해 이어서 읽는다.

안전한 주문 수집 순서는 다음과 같다.

1. `lastChangedFrom`과 저장된 `moreSequence`로 변경 목록 조회
2. 응답의 상품주문번호를 최대 300개로 묶어 상세 조회
3. 내부 저장 및 후속 처리를 트랜잭션으로 완료
4. 성공한 뒤에만 다음 커서를 저장
5. 겹치는 시간 구간을 재조회하고 상품주문번호로 중복 제거

### 정산

건별·일별 부가세와 정산, 수수료 상세 데이터를 제공한다. 주문 결제 금액과 상품주문 단위 합계가 항상 단순 일치한다고 가정하면 안 된다. 할인과 배송비 배분 규칙을 별도로 검증해야 한다.

### 통계

BizData 통계 API 일부는 브랜드스토어 전용이며 커머스솔루션마켓 구독이 필요하다. 공식 문서상 최대 18개월 전 데이터부터 제공되는 영역이 있다.

### N배송

SKU 목록·상세·연결상품 조회 API가 있다. 네이버 SKU ID와 판매자 상품/옵션 ID를 별도 키로 저장하고, 연결 관계를 주기적으로 동기화하는 편이 안전하다.

## 4. 호출 제한과 재시도

네이버 기술지원 안내에 따르면 토큰 버킷 방식의 호출 제한을 사용한다. 제한 계층에는 API 수용량, 애플리케이션별 Rate Limit, SELLER 토큰별 Quota Limit이 포함될 수 있다.

- `429`, `502`, `503`, `504`: 조회 요청만 지수 백오프로 제한 재시도
- `401 + GW.AUTHN`: 토큰 갱신 후 한 번 재시도
- `400`: 자동 재시도 금지. 입력 스키마와 오류 `message` 확인
- `403`: 권한, 약관 동의, 판매자/채널 상태, 허용 IP 확인
- 변경 요청: 멱등성이 문서로 보장되지 않으면 자동 재시도 금지

## 5. MCP 설계

이 구현은 두 층으로 나뉜다.

- 편의 도구: 계정, 채널, 상품, 카테고리, 주문, SKU의 대표 조회
- 범용 도구: 허용된 읽기 요청과 명시적으로 승인된 변경 요청

쓰기 작업은 기본 차단된다. 운영자가 환경 변수로 활성화하고, 호출자가 `EXECUTE`를 전달해야 한다. 이 이중 장치는 실수 가능성을 낮추지만 사용자 승인과 권한 통제를 대체하지 않는다.

## 6. 현재 구현 범위

지원:

- JSON 기반 GET/POST/PUT/PATCH/DELETE
- OAuth 토큰 캐시와 갱신
- 조회용 제한 재시도
- 추적 ID 보존
- 경로 고정 및 SSRF/경로 탈출 방지

미지원:

- multipart 상품 이미지 업로드
- 대용량 파일 다운로드/스트리밍
- 리뷰 및 톡톡 상담 수집
- 원격 MCP 인증 서버
- 네이버 문서 스키마 자동 생성
